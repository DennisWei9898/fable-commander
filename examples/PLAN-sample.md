# PLAN.md — 範例 state file（示範用）

> 這是一份**示範**：假想任務「建一個台股月營收爬蟲 + 資料品質檢核」跑完指揮官工作流 Phase 0–6 後，state file 的長相。實際內容會隨你的任務而變；重點是看**結構**——每個 phase 回寫、gate 有輪次紀錄、結尾有採納率。

## 版本

| 版本 | 時間點 | 修訂 |
|---|---|---|
| v0 | Phase 0 | 指揮官起草：3 phases、驗收草稿、6 條已知未知 |
| v1 | Phase 3 | 併入研究結果：資料源改用公開 API、驗收標準定稿 8 條 |

## 任務

抓取台股上市公司月營收（近 24 個月），落地成乾淨的 parquet，附資料品質檢核報告。

## 模型分工（使用者選定）

- 研究：Sonnet（Phase 1 選定）
- 執行 maker：Opus（Phase 4 選定）
- 驗收 verifier：Sonnet（獨立 fresh context，與 maker 跨模型）

## Phases × Gates

| Phase | 交付 | Gate（客觀可判） | 狀態 |
|---|---|---|---|
| P1 資料源接通 | fetch 模組 + 原始 JSON 快照 | 隨機抽 5 家公司 × 3 個月，與官方網頁數字一致 | ✅ 第 1 輪過 |
| P2 清洗落地 | parquet + schema | schema 驗證通過；列數 = 公司數 × 月份數 ±除牌名單 | ✅ 第 2 輪過 |
| P3 品質檢核 | QA 報告 | 缺值率 < 0.5%；重複鍵 = 0；月增率極端值逐筆有註記 | ✅ 第 1 輪過 |

## 已知未知（Phase 0 起草 → Phase 2 研究回答）

1. ~~官方 API 有沒有月營收 endpoint？~~ → 有，公開 API，免金鑰（來源已附 RESEARCH.md #1）
2. ~~歷史資料回溯上限？~~ → 該來源僅近 5 年（RESEARCH.md #2）
3. ~~除牌公司會不會造成缺列？~~ → 會，需另抓除牌名單對照（RESEARCH.md #4）
4. 頻率限制未寫明 → **仍未知**，列入風險節：實作採 1 req/s 保守節流

## 執行迴圈紀錄（Phase 5）

| Gate | 輪次 | verifier 判定 | 摘要 |
|---|---|---|---|
| P2 schema | 1 | ❌ FAIL | 3 家公司列數短缺——除牌對照表沒接上（附抽核清單） |
| P2 schema | 2 | ✅ PASS | 除牌名單併入後列數吻合；重跑驗證通過 |

## 人工閘紀錄

- 資料源需付費升級歷史區間 → 問過使用者，決定：不花錢，接受 5 年區間（v1 已記入）

## 收尾（Phase 6）

- **採納率**：3 項交付全數採納（3/3），QA 報告格式被使用者要求微調一次後接受
- **教訓回灌**：「除牌對照」這類邊界名單應在 Phase 0 就列入已知未知——已回寫 SKILL.md 提醒
